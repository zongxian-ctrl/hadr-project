Product Requirements · v1 · 2026-07-08 · distilled from REQS.md via the build-plan-product interview (QUESTIONS.md, CONTEXT.md, ADRs in docs/adr/)

Problem

Disaster-relevant signal is scattered across three feeds that are different in kind — raw sensor data (USGS), modeled multi-hazard alerts (GDACS), and human-curated humanitarian reporting (ReliefWeb) — and buried under hundreds of routine, duplicate, and later-revised records. A person who wants morning situational awareness must watch three sites, reconcile the same physical event appearing under three identifiers, and notice when yesterday's facts were quietly revised. The four questions that matter are always the same: what happened, where, how bad, and who is affected.

Product

An autonomous agent that publishes a terse daily situation report to dashboard.html at 08:30 Asia/Singapore, covering every GDACS Orange / Red event worldwide, enriched with USGS detail and ReliefWeb humanitarian context, with explicit corrections when previously reported facts change — and silence on mornings when nothing did.

Audience & success

  • Reader: the project owner, personally — a 2-minute morning scan. Terse entries, no boilerplate.
  • Success bar (v1): a trustworthy pipeline that runs unattended on schedule, stays quiet when the world is quiet, and never silently loses a correction.
  • Stretch: a report worth keeping after the course week.

User stories

Morning scan
As the reader, I open dashboard.html after 08:30 and within two minutes know every currently significant event: what, where, how bad, who is affected — each with its GDACS alert colour and links to sources.
Quiet mornings
As the reader, on a no-change morning I see yesterday's report unchanged with a fresh "checked at" stamp — and no model was invoked to conclude that.
Corrections
As the reader, when an event I already read about is revised, upgraded, downgraded, or deleted upstream, the next report calls it out in an Updates & Corrections section rather than silently rewriting history.
One event, one entry
As the reader, I see one entry per real-world event with all three feeds' records attached — never the same earthquake three times under different identifiers, and never two genuinely distinct earthquakes merged into one.
Feed liveness
As the reader, when a feed was unreachable the report tells me so; an empty report never means "the agent couldn't look".
Sequences
As the reader, an aftershock sequence appears as one line under its mainshock, not as dozens of entries.
Recent history
As the reader, I can scan the last 7 days of reports from the same page.

Solution overview

08:30 SGT (GitHub Actions, cron) │ ├─ 1. DETERMINISTIC STEP (scripts/, Python — never calls a model) │ fetch GDACS · USGS · ReliefWeb (conditional GET, fixtures in tests) │ normalise → unified events (GLIDE join, else type+space+time) │ diff vs state/seen-events.json (content + alert level, not bytes) │ exit code: CHANGED | UNCHANGED | DEGRADED(feed down) │ ├─ 2. UNCHANGED → touch "checked at", stop. (no model call) │ └─ 3. CHANGED → headless Claude runs /sitrep skill writes dashboard.html (report + corrections + feed status + 7-day history) commits dashboard.html + state/ in one commit

The split is the architecture: the model never decides whether to wake up. Everything that must give the same answer twice is deterministic Python in scripts/; judgement — severity narrative, writing the report — lives in the /sitrep skill.

Event model

The feeds' ontologies differ (a USGS record is one rupture; a GDACS event is a multi-episode alert; a ReliefWeb disaster is a humanitarian situation spanning possibly several ruptures), so records relate many-to-many. The canonical record is the unified event, keyed on GDACS eventid: physical events (USGS) and situations (ReliefWeb) link into it. Linking uses the GLIDE number when both sides carry it, and falls back to type + distance + time-window correlation tuned not to merge distinct near-simultaneous events (the Venezuela M7.1/M7.5 doublet is the canonical counter-example). Linking re-runs every cycle — GLIDE numbers and ReliefWeb records arrive days late.

Triggering & severity

GDACS alert colour is the sole trigger: Orange and Red events enter the report; Green is suppressed. Severity is therefore modeled human impact by construction — never raw magnitude. USGS PAGER and felt data enrich earthquake entries. Events GDACS under-scores at first are an accepted miss class: they surface via the corrections mechanism when upgraded.

Functional requirements

F1Ingest three feedsGDACS GeoJSON event list; USGS GeoJSON summary + FDSN updatedafter for revisions; ReliefWeb disasters RSS (API behind same interface post-approval). Conditional GET everywhere it's honoured.
F2NormaliseAll times to UTC (GDACS timestamps are unlabeled UTC — parse explicitly); GeoJSON coordinates are [lon, lat, depth]; store USGS's full ids alias set.
F3UnifyUnified event view keyed on GDACS eventid; GLIDE join, correlation fallback; aftershocks clustered under mainshock; re-linkable every run.
F4FilterOrange/Red only, global, all GDACS rapid-onset hazard types. Configurable --min-alert for testing/demo.
F5Detect changeDiff event content and alert level vs committed state — never feed bytes. "Changed" = new qualifying event · level change · material revision · upstream deletion · feed-liveness transition.
F6Gate the modelDeterministic step exits with a status the workflow branches on; the model runs only on CHANGED/DEGRADED.
F7Report/sitrep skill writes dashboard.html: entries (what/where/how bad/who affected), Updates & Corrections, per-feed status line, 7-day history. Single column, alert-colour coded, light/dark.
F8PublishOne commit per changed morning: dashboard.html + state/. No commit on quiet mornings.

Testing decisions

  • Fixtures, never live feeds. The deterministic step is tested against recorded feed payloads (replay mode); Orange+ events are too rare to test against the live world, and live tests aren't repeatable anyway.
  • Golden-file tests for the change detector: given state N and fixture N+1, assert the exact CHANGED/UNCHANGED/DEGRADED verdict and diff — including the trap cases: byte churn with no content change, a revision, a downgrade, a deletion, a feed outage.
  • Correlation unit tests include the doublet case (two real events close in space/time must not merge) and the alias case (one event, several USGS IDs, must not split).
  • Demo path: --min-alert green plus fixtures makes the pipeline demonstrable on any day, regardless of the world.

Out of scope (v1)

  • Intra-day or near-real-time alerting — daily 08:30 only.
  • Slow-onset and conflict crises as triggers (ReliefWeb remains enrichment; revisit when it can trigger).
  • Forecasting or prediction of any kind.
  • Feeds beyond GDACS / USGS / ReliefWeb.
  • Acting on alerts (notification fan-out, dispatch, tasking) — monitoring only.

Risks & open items

  • Orange+ rarity vs a 3-day course: a live demo may see zero qualifying events — mitigated by fixtures + --min-alert (see CONTEXT.md, flagged tension).
  • GDACS EVENTS4APP is an unversioned internal API — parse defensively; RSS is the fallback contract.
  • ReliefWeb RSS never republishes updates — situation status is stale until the API appname is approved (requested, in flight).
  • USGS rolling window drops late revisions — revision tracking must use the FDSN updatedafter query, not the summary feed.
  • Early magnitudes for great earthquakes run low — the corrections mechanism is the safety net, not a reason to trust first numbers.
